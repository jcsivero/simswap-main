Instalar Internet Information Server Versión 4.0

  • Windows NT 4.0 y Service Pack 3 El servicio Servidor de Windows NT® debe estar instalado y ejecutarse.
  • Al instalar IIS 4.0 en un equipo que ejecuta Windows NT Server, los servicios Inicio de sesión y Examinador de equipos deben estar en ejecución. Si no se ejecutan estos servicios, verá un cuadro de diálogo que dice "No se puede detectar el tipo de sistema operativo" y el programa de instalación fallará.
  • Quitar el software Alpha, Beta 1, o Beta 2 Si tiene IIS versión 4.0 Alpha o IIS versión 4.0 (Beta 1 o Beta 2), asegúrese de desintalar esa versión (antes de instalar la versión actual). Para ello, en primer lugar inicie el programa de instalación del disco compacto que contiene la versión anterior o bien, en Windows NT, haga clic en Inicio, seleccione Programas, Microsoft Internet Information Server (Común) y, a continuación, haga clic en Instalación de Internet Information Server. Haga clic en Siguiente y, a continuación, haga clic en Quitar todo.

    • Se producirán errores de instalación de ODBC durante el programa de instalación si hay cualquier otra aplicación o servicio del sistema en ejecución que utilice el software de ODBC existente en el sistema. Si ocurre algún error de ODBC, detenga todas las aplicaciones de escritorio y de servicios del sistema, y ejecute de nuevo el programa de instalación.

    Notas

    Windows NT 4.0 Option Pack sólo se puede instalar en Windows NT Server 4.0 y Windows NT Workstation 4.0.

    Personal Web Server para Windows95, incluido en Windows NT 4.0 Option Pack, se instala tanto en Windows 95 como en la versión Beta 3 de Windows 98.

    Windows NT 4.0 Option Pack no se puede instalar en ninguna versión de Windows NT 5.0.

Otras consideraciones de instalación

Reinstalar Windows NT Service Pack 3
Si reinstala Windows NT Service Pack 3 por cualquier razón después de instalar IIS 4.0, no sobrescriba los nuevos archivos que instaló IIS 4.0.
Desinstalar el producto
Para desinstalar el producto, vuelva a ejecutar el programa de instalación de esta versión. Hay varias formas de tener acceso al programa de instalación Puede utilizar el disco compacto o hacer clic en Inicio, seleccionar Programas, Windows NT Option Pack 4.0 y, a continuación, hacer clic en Instalación de Windows NT Option Pack 4.0. Siga las instrucciones de los cuadros de diálogo y, cuando se le pida, haga clic en el botón Agregar o quitar para desinstalar el software.
Errores y problemas de instalación
  • Los errores que ocurren durante la instalación desatendida se escribirán en el Registro de Windows NT. Vaya a la siguiente ruta de acceso del Registro para obtener información acerca del error:
    HKEY_LOCAL_MACHINE\
     Software
      \Microsoft
       \Windows
        \Setup
         \OCmanager
          \Errors
  • Al instalar IIS 4.0 en un controlador de reserva (BDC) de Windows NT, debe seleccionar la opción Instalación personalizada en el programa de instalación y deshabilitar Index Server y el sitio World Wide Web de ejemplo. Si desea instalar Index Server o el sitio World Wide Web de ejemplo en un BDC, utilice la característica Agregar o quitar programas del Panel de control para instalar estos componentes después de que se haya completado la instalación de IIS.

Consideraciones de actualización

  • La versión 4.0 de IIS no es compatible con Microsoft Proxy Server 1.0. Para poder utilizar Proxy Server con IIS, actualice Proxy Server de la versión 1.0 a la versión 2.0. Proxy Server versión 2.0 se debe instalar o reinstalar después de que se actualice IIS a la versión 4.0. Además, las instalaciones actuales de Proxy Server 2.0 no funcionarán correctamente después de que se instale IIS 4.0. Debe volver a ejecutar el programa de instalación de Proxy Server 2.0 tras instalar IIS4.0.
  • Si actualiza desde una versión anterior de IIS y tiene problemas para ver Default.asp, es probable que tenga una versión más antrigua de este archivo. Para instalar la nueva versión IIS 4.0 de este archivo, debe quitar manualmente la versión antigua de Default.asp. La ubicación predeterminada es C:\InetPub\Wwwroot.
  • No puede utilizar el Administrador de servicios Internet de Internet Information Server 3.0 para configurar propiedades administrativas de Internet Information Server 4.0.
Problemas de Certificate Server
Si tiene instalado Certificate Server, debe detener este servicio antes de instalar IIS. Si no lo hace, no se instalará n correctamente los controladores actualizados de ODBC. En general, se recomienda cerrar todas las aplicaciones de fondo antes de ejecutar el programa de instalación para evitar problemas.

Para detener el servicio Certificate Server en Windows NT, haga clic en Inicio, seleccione Configuración y haga clic en Panel de control y ejecute la aplicación Servicios. Seleccione Certificate Authority y haga clic en el botón Detener. Para reiniciar el servicio una vez realizada correctamente la instalación de IIS, abra el Panel de control, ejecute la aplicación Servicios, seleccione Certificate Authority y haga clic en el botón Iniciar.

Si recibe un error durante la instalación que indica que no se cargaron Msdasql.dll u Odbc*.dll, tendrá que cancelar la instalación, detener los servicios y reiniciar.

Instalar.exe no tiene firma de código.
Al ejecutar la instalación desde el CD de IIS mediante la página Web de instalación, Internet Explorer mostrará un mensaje de error que indica que Instalar.exe no tiene firma de código. Instalar.exe es un archivo de 16 bits y los formatos de archivo de 16 bits no pueden tener firma de código.

Documentación del producto

Toda la documentación del producto está disponible en pantalla. Puede tener acceso a los temas de Ayuda si hace clic en Inicio y selecciona el grupo de programas donde está instalado el producto o mediante el menú Ayuda de la herramienta de administración. También hay disponible Ayuda contextual si hace clic en el botón Ayuda en una hoja de propiedades o cuadro de diálogo.

Utilizar la documentación

Tenga en cuenta las siguientes instrucciones al utilizar la documentación en pantalla:

  • El servicio WWW debe estar instalado y se debe ejecutar el servidor para poder ver la documentación (con la excepción de las notas de la versión y los archivos de solución de problemas). Si el sitio Web está detenido, al hacer clic en Documentación del productoobtendrá el mensaje "No se ha podido establecer una conexión con el servidor". Si se produce este error, inicie el Administrador de servicios Internet y compruebe el estado del sitio Web; si está detenido, inicie el servicio.
  • En Internet Explorer 4.01 los usuarios deben habilitar la opción No usar servidor proxy para direcciones locales (intranet) de la hoja de propiedades Conexión para ver los archivos locales, como la documentación del producto.
  • Si hace clic en el icono Imprimir de la ficha Contenido de la documentación de Option Pack y selecciona la opción Imprimir todo lo que se encuentra bajo el encabezado actual, es posible que se produzca una disminución momentánea del rendimiento del servidor si la sección seleccionada contiene un gran número de temas.

    • Las páginas impresas mediante el icono de impresión de Contenido no incluirán gráficos ni documentos vinculados. Además, el icono de impresión de Contenido no funciona junto con las secuencias de comandos y los programas de ejemplo que se encuentran en la sección Ejemplos para el programador de la documentación de IIS. Si selecciona un tema de la sección Ejemplos para el programador y hace clic en el icono de impresión de Contenido, aparecerá un mensaje de error que indica que no se puede encontrar el archivo.

  • Para ver la documentación de IIS de forma remota, utilice la siguiente dirección URL: http://<nombre_servidor>/iisHelp/, donde <nombre_servidor> es el nombre del servidor que ejecuta IIS.

Generalidades

    Administración

  • En versiones anteriores de IIS, podía detener todo el servicio Web si escribía en la línea de comandos net stop w3svc. Esto finalizaba todos los servicios Web del equipo y cerraba el proceso Inetinfo.exe. Como resultado de la nueva arquitectura de múltiples sitios Web, hay otro servicio, Iisadmin, que continúa su ejecución, incluso cuando se cierra W3SVC. Ocurre lo mismo al iniciar y detener servicios desde el Panel de control. Si realmente desea descargar el proceso Inetinfo.exe y asegurarse de que todas las extensiones se han descargado correctamente, debe escribir net stop iisadmin, en lugar de net stop w3svc. Escribir net start w3svc o net start msftpsvc inicia automáticamente Iisadmin.
  • Al iniciar el servidor Web, éste migra todas las nuevas raíces virtuales del Registro a la metabase ("nueva" significa que no está todavía en la metabase). A continuación, el servidor elimina todas las raíces virtuales del Registro y refleja las raíces virtuales de la metabase en las del Registro. Cualquier cambio de las raíces virtuales de la metabase se vuelve a reflejar en las raíces virtuales del Registro, lo que elimina las entradas del Registro que se cambiaron o agregaron desde la instalación original.
  • Se debe ejecutar el servicio PStore para poder ejecutar IIS.
  • Si tiene un archivo .dll con dependencias especiales que requieren que estos archivos se carguen antes que otros procesos de servidor Web, puede crear en el Registro una lista definida de los archivos .dll que se tienen que cargar previamente en el programa Inetinfo.exe antes de que se inicie ningún otro servicio de servidor Web.

    • Para cargar previamente archivos .dll, cree una nueva clave del Registro llamada PreloadDlls en la siguiete ruta de acceso del Registro:

    HKEY_LOCAL_MACHINE\
     SYSTEM
      \CurrentControlSet
       \Services
        \Inetinfo
         \Parameters

    Defina esta entrada como de tipo multi-sz, con datos que consisten en los nombres de los archivos .dll que desea cargar previamente (separados por retornos de carro). Si los archivos residen en la misma ruta de acceso, puede escribir únicamente el nombre del archivo .dll. Si el archivo .dll reside en otro directorio, escriba el nombre del archivo y su ruta de acceso completa.

  • Cuando intente hacer copia de seguridad de la información de configuración de un sitio Web se le pedirá que escriba una frase de comentario antes de guardar la información de configuración. Sin embargo, no puede guardar esta información si el comentario contiene caracteres no alfanuméricos, incluidos los caracteres no imprimibles o cualquiera de los siguientes caracteres:

    • / \ \ * . ? \ "& ! @ # $ % ^ ( ) = + | ` ~ 

  • No puede utilizar el Administrador de servicios Internet para administrar de forma remota un equipo que ejecuta IIS 4.0 Beta 3.0 desde un equipo que ejecute la versión final de IIS 4.0. Del mismo modo, no puede administrar un equipo que ejecute la versión final de IIS 4.0 desde uno que ejecute IIS 4.0 Beta 3.0. Esto se aplica a todas las demás aplicaciones que utilizan la interfaz AdminBase. La solución consiste en poner la versión comercial del archivo Admwprox.dll en todos los equipos y, a continuación, reiniciar.

    • Administración remota con el Administrador de servicios de Internet (HMTL)

  • Para administrar un equipo remoto que ejecuta IIS 4.0 mediante el Administrador de servicios Internet (HTML) debe iniciar la sesión con una cuenta válida de administrador (u operador) para ese equipo remoto. Con versiones anteriores podía crear una sesión en \\NombreServidor\IPC$ si utilizaba una cuenta de usuario diferente para obtener acceso; esto no funcionará en IIS 4.0. Si desea tener acceso al Administrador de servicios Internet (HTML) en un equipo remoto con una cuenta diferente (es decir, una cuenta distinta de la que utiliza para iniciar la sesión en el equipo local), debe habilitar la autenticación Básica y deshabilitar la autenticación Desafío/Respuesta de Windows NT en la hoja de propiedades Seguridad de directorios, de forma que se le pidan las credenciales adecuadas de cuenta de usuario.
  • Para que el Administrador de servicios de Internet (HTML) funcione correctamente, debe establecer la memoria caché del explorador para que se actualice en cada visita a una página HTML. En IE 4.01 puede establecerlo mediante los siguientes pasos:
    1. Seleccione el elemento de menú Ver.
    2. Seleccione Opciones de Internet .
    3. En la ficha General, haga clic en el botón Configuración.
    4. Bajo Comprobar si hay nuevas versiones de las páginas guardadas, seleccione la opción Cada vez que se visita la página.

  • Algunos directorios virtuales de Internet Information Server están configurados de tal forma que aparecen como directorios no virtuales para los Objetos de administración de IIS. Esta configuración evita que los Objetos de administración se administren mediante Administrador de servicios de Internet (HTML). Los directorios virtuales afectados son los siguientes:

    • Iishelp/Ado
    Iishelp/Adc
    Iishelp/Msadc
    Iishelp/Ics
    Msadc
    Iishelp/Sse/Sa
    Iishelp/Sse/Uaexpress
    Iishelp/Ics/I_cmak
    Iishelp/Ics/I_cps
    Iishelp/Ics/I_ias
    CertSrv
    CertQue
    CertAdm
    CertControl

    Una secuencia de comandos llamada Fixcfg.vbs (que se encuentra de forma predeterminada en Winnt\System32\Inetsrv\Adminsamples) corregirá este problema.

  • Al administrar el servidor Web con el Administrador de servicios Internet (HTML) a través de una conexión de proxy, no puede utilizar la autenticación Desafío/Respuesta de Windows NT. Los servidores proxy sólo permitirá n al servidor Web utilizar la autenticación Básica.
  • Si crea un nuevo sitio Web en Microsoft Management Console, los Operadores del sitio Web no pueden conectar de forma remota a ese sitio con el Administrador de servicios Internet (HTML) hasta que cree un directorio virtual llamado IISADMIN. (Observe que los usuarios designados como Administradores pueden administrar de forma remota el sitio a través del sitio Web administrativo sin este directorio virtual). Si crea el sitio Web mediante el Administrador de servicios de Internet (HTML), el servidor Web creará automáticamente el sitio IISADMIN porque el servidor supone que deseará administrar ese sitio de forma remota.

    • Para crear el directorio virtual IISADMIN:

    1. En el Administrador de servicios Internet, seleccione el sitio Web que desea que los Operadores administren de forma remota.
    2. Haga clic con el botón secundario del mouse (ratón), seleccione Nuevo y, a continuación haga clic en Directorio virtual.
    3. En el cuadro Alias, escriba IISADMINy, a continuación, haga clic en Siguiente.
    4. Escriba <LetraUnidad>:\Winnt\System32\Inetsrv\Iisadmin donde se reemplaza LetraUnidad con la letra de la unidad local. (Esta ruta de acceso física es la misma que la ruta de acceso del directorio Iisadmin del sitio Web predeterminado.) Haga clic en Siguiente.
    5. Seleccione Permitir acceso de Lectura y Permitir acceso a archivos de comandos. Haga clic en Siguiente.
    6. Haga clic en Finalizar.
    7. Seleccione el nuevo directorio virtual IISADMIN para el sitio Web y abra sus hojas de propiedades.
    8. Seleccione la hoja de propiedades Seguridad de directorios y bajo Acceso anónimo y Control de autenticación, haga clic en Modificar.
    9. Desactive la casilla de verificación Permitir acceso anónimo.
    10. Seleccione Autenticación básica o Autenticación Desafío/Respuesta de Windows&nbs;NT y haga clic en Aceptar.

    Interfaz de programación de aplicaciones de servidor Internet (ISAPI)

  • La propiedad InProcessIsapiApps (MD_IN_PROCESS_ISAPI_APPS) de la metabase de configuración ahora permite entradas de nombres de .dll sin rutas de acceso que los cualifiquen. Una entrada sin una ruta de acceso que la cualifique coincidirá con cualquier petición de ISAPI que contenga el nombre de la .dll, cualquiera que sea la ruta de acceso física que se especifique en la petición. Por ejemplo, lo siguiente sería una lista válida para la propiedad: "D:\Winnt\System32\Inetsrv\Ssinc.dll" "D:\Winnt\System32\Inetsrv\Httpodbc.dll" "Author.dll" "Shtml.dll" "Admin.dll".
  • Al depurar aplicaciones ISAPI, filtros ISAPI, componentes de servidor u otro código que escriba para que se ejecute sobre IIS, puede ser útil ejecutar IIS como un proceso. La información acerca de cómo configurar IIS 4.0 para que se ejecute como un proceso se describe en la documentación del producto. Se han agregado al SDK de IIS 4.0 instrucciones adicionales y dos archivos .reg que automatizan el proceso de cambiar IIS para que se ejecute como un servicio o como un proceso.

    • Para instalar el SDK

    1. En el cuadro de diálogo Instalación de IIS, seleccione Internet Information Server y haga clic en Mostrar subcomponentes.
    2. Seleccione Documentación, haga clic en Mostrar subcomponentes y, a continuación, haga clic en SDK.
    3. Haga clic en Aceptar.

    Puede encontrar las instrucciones en Inetpub\Iissamples\Sdk\Components\IIS_as_process.txt. Para obtener más información, el tema "Establecer un entorno de depuración" en la sección Desarrollo de aplicaciones Web avanzadas de la documentación de Internet Information Server describe cómo establecer un entorno para depurar extensiones ISAPI.

  • El identificador de la metabase MD_ALLOW_PATH_INFO_FOR_SCRIPT_MAPPINGS no está documentado en Metabase Identifier Reference de Programmer's Reference. La documentación para la correspondiente propiedad de metabase de automatización, AllowPathInfoForScriptMappings, en Administration Property Reference, también se aplica a MD_ALLOW_PATH_INFO_FOR_SCRIPT_MAPPINGS. El tipo de usuario es IIS_MD_UT_SERVER.
  • En Metabase Identifier Reference, que forma parte de la documentación Programmer's Reference para el identificador de metabase MD_SCRIPT_MAPS, la constante MD_SCRIPTMAPFLAG_SCRIPT se identifica incorrectamente como MD_SCRIPTMAPFLAG_SCRIPT_ENGINE.
  • Las dos propiedades de metabase de configuración, AppAllowDebugging y AppAllowClientDebug, se pueden establecer en la clave IIsWebDirectory, además de en las otras claves especificadas en la documentación.

    Páginas Active Server

  • Las propiedades definidas para los objetos Request, Response y Server de ASP sólo son válidas durante el proceso de una petición. Por ejemplo, si define una propiedad en Global.asa, como Response.Buffer = True, la propiedad sólo se aplicará a una única petición y no a todas las peticiones.
  • Si detiene el servicio WWW desde el Administrador de servicios Internet mientras depura una secuencia de comandos, el Administrador de servicios Internet no funcionará hasta que continúe la ejecució n de la secuencia de comandos que depura. Normalmente, si detiene el servicio Web mediante el Panel de control o el comando net stop, las secuencias que se depuran con Microsoft Script Debugger finalizan. Sin embargo, si detiene un sitio Web desde el Administrador de servicios Internet, el sitio detendrá su funcionamiento si está depurando actualmente secuencias de comandos que se ejecutan en el servidor que detiene. Si va a Microsoft Script Debugger y selecciona Detener depuración para las secuencias de comandos que está depurando, el Administrador de servicios Internet seguirá funcionando.
  • Actualmente hay problemas conocidos al utilizar PerlScript con ASP. Si ejecuta PerlScript de ActiveWare, póngase en contacto con ActiveWare para obtener un nuevo motor de PerlScript.
  • En la documentación de ASP se indica incorrectamente que el método MapPath no es compatible con la sintaxis de rutas de acceso relativas, como ./ruta de acceso o ../ruta de acceso. El método MapPath admitirá sintaxis de rutas de acceso relativas sólo si la propiedad EnableParentPaths de la metabase está habilitada. Cuando la propiedad EnableParentPaths no está habilitada, utilizar el método MapPath con una ruta de acceso relativa produce un error.
  • Al definir permisos de Windows NT para un directorio para el que piensa utilizar autenticación anónima o de Desafío/Respuesta de Windows NT, debe asignar permisos de Lectura a la cuenta de usuario anónimo (IUSR_ nombre_equipo) o a la cuenta propietaria de la aplicación (IWAM_nombre_equipo) si el directorio contiene una aplicación fuera de proceso y un archivo Global.asa. No definir el permiso de Lectura puede dar como resultado que ASP pase por alto el archivo Global.asa.

    • De forma similar, para aplicaciones en proceso contenidas en directorios almacenados en una unidad de red, debe asegurarse de que la cuenta de usuario anónimo tiene permisos de Lectura de Windows NT.

  • En MicrosoftVisual Basic o VBScript, si llama a GetObject() para crear una instancia de un Objeto Admin de IIS y GetObject falla, el código de error devuelto se cambia en ole32 de MkParseDisplayName() a un error MK_E_SYNTAX.
  • Si crea o utiliza componentes desarrollados en Microsoft® Visual Basic 5.0 (Edición empresarial o Edición profesional) para utilizarlos con ASP, se recomienda encarecidamente que actualice a Visual Studio#153; 97 Service Pack 2, tanto para los equipos de programación como para los servidores.

    • Puede descargar Visual Studio 97 Service Pack 2 en http://www.microsoft.com/vstudio/sp/ (este sitio está en inglés).

  • El objeto Scripting.Dictionary está marcado erróneamente como procesamiento Ambos. Se debería marcar como subprocesamiento Controlado. Para cambiar esto, utilice el Editor del Registro para abrir la siguiente clave del Registro:

    • HKEY_CLASSES_ROOT\
      CLSID
       \{EE09B103-97E0-11CF-978F-00A02463E06F}
        \InprocServer32

    Cambie el valor con nombre para ThreadingModel a Apartment. Si utiliza el objeto Dictionary con alcance de aplicación sin hacer este cambio, pueden dañarse los datos.

    Seguridad de servidor

  • Aunque el Administrador de servicios Internet le permitirá habilitar la sincronización de contraseñas para una cuenta de usuario anónimo no local, está configuración no es compatible y puede provocar que se deniegue todo acceso al servidor Web o FTP. La sincronización de contraseñas sólo se debe utilizar con cuentas de usuario anónimo definidas en el equipo local. Si el sitio Web se basa en permitir que el usuario anónimo pueda tener acceso a los recursos de red, no puede utilizar la característica de sincronización de contraseñas de autenticación anónima y debe especificar una contraseña válida para el usuario anónimo.

    • Problemas del sitio de ejemplo Exploration Air

  • Si no ha instalado Remote Data Service, incluido con Microsoft Data Access Components 1.5, necesitará modificar el archivo Prefs.txt (que se encuentra de forma predeterminada en Inetpub\IISSamples\ExAir\Benefits\Prefs.txt) para deshabilitar la característica Remote Data Service, que se utiliza en la aplicación Benefits de Exploration Air. Específicamente, cambie RDSAvailable de True a False.
  • Al utilizar la característica Secure Sockets Layer (SSL) para cargar archivos, el componente Posting Acceptor cargará correctamente un archivo en el servidor Web; sin embargo, la redirección a Verify.asp fallará. Esto no afectará al funcionamiento del componente Posting Acceptor.
  • La base de datos de Microsoft SQL Server del sitio de ejemplo Exploration Air está configurada para utilizar el Id. de usuario (UID) y la contraseña predeterminados (es decir, UID=sa y sin contraseña). Otras configuraciones de SQL no funcionarán.
  • El componente de carga de archivos que se encuentra en el vínculo Business Partners Only no funcionará si el directorio para la carga de archivos no está correctamente configurado. A continuación se ofrece un procedimiento que puede utilizar para configurar el sistema para cargar archivos:
    1. En el directorio IISSamples\ExAir\BusinessPartners, cree un subdirectorio llamado Upload.
    2. Haga clic en Inicio, seleccione Programas, Windows NT 4.0 Option Pack, Internet Information Server y, a continuación, haga clic en Administrador de servicios Internet.
    3. En el panel izquierdo del Administrador de servicios Internet, haga clic en la carpeta Internet Information Server y, a continuación, haga doble clic en el icono llamado como el servidor.
    4. Haga doble clic en Sitio Web predeterminado y explore hasta el directorio virtual IISSamples/ExAir/Business Partners/Upload.
    5. Haga clic con el botón secundario del mouse en Upload y seleccione Propiedades. En la hoja de propiedades Directorio, bajo Permisos de acceso, seleccione Escritura.
    6. Bajo Permisos en Configuración de la aplicación , seleccione Ninguno. Esta configuración permitirá que el componente Posting Acceptor funcione correctamente.
    7. Cierre el Administrador de servicios Internet y vuelva a intentar la carga de archivos.

    Generalidades

  • Además de definir la versión de la configuración del servidor Web, la característica de copia de seguridad y restauración puede utilizarse para restaurar una configuración anterior de IIS cuando ocurre un problema que se puede corregir si se desinstala y reinstala IIS. Sin embargo, este proceso no funcionará cuando se necesite una reinstalación completa del sistema operativo. Si el equipo encuentra un problema que requiere que vuelva a instalar Windows NT 4.0 Option Pack, utilice el siguiente procedimiento para restaurar una configuración anterior:
    1. En el panel izquierdo del Administrador de servicios Internet, haga clic en la carpeta Internet Information Server y, a continuación, seleccione el icono con el nombre del equipo.
    2. Desde la barra de herramientas del Administrador de servicios Internet, haga clic en Acción y seleccione Hacer copia de seguridad o restaurar configuración.
    3. En el cuadro de diálogo Hacer copia de seguridad o restaurar la configuración, haga clic en Crear copia de seguridad .
    4. En el cuadro de diálogo Nombre de copia de seguridad de la configuración, escriba un nombre para el archivo de copia de seguridad. De forma, el archivo de copia de seguridad se almacena en el directorio C:\Winnt\System32\Inetsrv\Metaback De forma alternativa, el archivo de copia de seguridad se puede almacenar en el directorio especificado en la siguiente entrada del Registro (si ha creado esta entrada):

      5. HKEY_LOCAL_MACHINE\
       SOFTWARE
        \Microsoft
         \InetStp
          \MetaBackup

      Seleccione el valor BackupPath, que contiene el nombre del archivo de copia de seguridad que creó con el Administrador de servicios Internet.

    5. Copie el archivo a una ubicación fuera del subdirectorio Inetsrv. Los archivos de copia de seguridad del subdirectorio Metaback se eliminan durante la desinstalación.
    6. Desinstale y reinstale Windows NT 4.0 Option Pack.
    7. Copie el archivo de copia de seguridad al subdirectorio Metaback (de forma predeterminada C:\Winnt\System32\Inetsrv\Metaback\).
    8. En el cuadro de diálogo Nombre de copia de seguridad de la configuración, seleccione el archivo de copia de seguridad que creó y, a continuación, haga clic en Restaurar. Aunque la restauración puede fallar, se restaurará parte de la configuración de la que hizo copia de seguridad.
    9. Escriba en un símbolo del sistema: cscript.exe C:\Winnt\System32\Inetsrv\Adminsamples\Adsutil.vbs enum w3svc. Este comando enumerará varias configuraciones del servicio Web de IIS. A partir de las configuraciones enumeradas, encuentre los valores de WamUserName y WAMUserPass. Estos valores representan la Id. de usuario y contraseña que Windows Application Manager (WAM) utiliza para iniciar aplicaciones fuera de proceso. Durante la reinstalación se restablece la contraseña en el Administrador de usuarios.

      Nota   Para emplear la utilidad Adsutil.vbs debe haber instalado Windows Scripting Host como parte de la instalación de Windows NT 4.0 Option Pack.

    10. Haga clic en Inicio, seleccione Herramientas administrativas y haga clic en Administrador de usuarios para dominios. Haga doble clic en la cuenta de usuario IWAM_<nombre_equipo>. Escriba la contraseña recuperada en el paso anterior.
    11. Haga clic en Aceptar.
    12. En el cuadro de diálogo Nombre de copia de seguridad de la configuración, seleccione el archivo de copia de seguridad que creó y, a continuación, haga clic en Restaurar. La configuración debe estar completamente restaurada.
