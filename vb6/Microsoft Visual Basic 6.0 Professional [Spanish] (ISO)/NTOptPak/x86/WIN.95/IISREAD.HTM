Bienvenidos a Personal Web Server 4.0

Bienvenidos a Personal Web Server. En el paquete de Personal Web Server, Microsoft ha unido nuevas funcionalidades que hacen de su existente sistema operativo Microsoft® Windows  95® o posterior una plataforma aún más fácil de utilizar para compartir información en un sitio Web personal.

El paquete de Personal Web Server simplifica la instalación de cualquiera de las siguientes características nuevas:

Microsoft Personal Web Server 4.0
Un servidor Web personal que puede utilizarse para alojar un sitio Web en la intranet corporativa o para desarrollar y probar un sitio Web antes de alojarlo en un proveedor de servicios Internet (ISP).
Microsoft Internet Explorer 4.01
La versión más reciente de Microsoft® Internet Explorer proporciona las características esenciales que necesita para que buscar, explorar y comunicarse a través de Internet sean una parte sencilla de la utilización de su PC.
Extensiones de servidor de Microsoft FrontPage
Permite el uso de Microsoft® FrontPage® para administrar el sitio Web, así como crear el contenido del sitio.
Microsoft Transaction Server 2.0
Permite la creación de aplicaciones de Microsoft® Transaction Server (MTS). Las transacciones son operaciones del servidor que se ejecutan correctamente o fallan en su totalidad, incluso aunque las operaciones consten de varios pasos. MTS también admite el aislamiento del proceso de las aplicaciones.
Microsoft Data Access Components 1.5
Utilización sencilla de bases de datos con compatibilidad para ActiveX™ Data Objects (ADO) y el controlador Microsoft® Access.
Internet Connection Services para el Servicio de acceso remoto de Microsoft
Un conjunto de servicios que facilitan la creación de redes privadas virtuales (VPN) y conexiones telefónicas mejoradas.
Microsoft Message Queue Client
Permite a las aplicaciones enviar notificaciones de transacciones y continuar el proceso sin esperar la confirmación de que la transacción se ha completado.

Comparación de productos: PWS para Windows 95 y PWS para Windows NT Workstation

PWS para Windows 95 proporciona una interfaz gráfica de administración diseñada para usuarios sin experiencia previa en la creación y administración de sitios Web. Esta interfaz se proporciona también como la herramienta predeterminada de administración en Personal Web Server para Microsoft® Windows NT® Workstation; sin embargo, con la versión de Windows NT Workstation también puede administrar su sitio personal de publicación con el Administrador de servicios Internet, la misma herramienta de administración con características completas que se utiliza para controlar Microsoft Internet Information Server.

Notas   Los componentes de Windows NT 4.0 Option Pack sólo se pueden instalar en equipos que ejecutan Windows NT Server 4.0, Windows NT Workstation 4.0 o Windows 95. Windows NT 4.0 Option Pack no se puede instalar en equipos que ejecutan la versión Beta 1.0 de Windows NT 5.0.

Para instalar Personal Web Server
  1. No es necesario que quite las versiones anteriores de PWS antes de instalar esta versión, a menos que haya instalado las versiones Alpha o Beta 2 del producto. Para quitar las versiones Alpha o Beta 2, utilice el botón Quitar todo del programa de instalación de Personal Web Server. También es necesario quitar la versión anterior de Winsock 2 instalada con las versiones Alpha y Beta 2.

    2. Vea Solución de problemas para obtener más información.

  2. Seleccione el vínculo Personal Web Server e instale PWS.
  3. Reinicie el equipo para seguir con el proceso de instalación. Si está realizando la instalación a través de Internet, seleccione la plataforma y siga las instrucciones de la pantalla.

    4. Importante    La instalación de PWS necesita que Microsoft Internet Explorer versión 4.01 esté instalado en el equipo. Si elige no instalar Internet Explorer 4.01, cancelará la instalación de PWS. Además, si quita o reinstala Internet Explorer 4.01 después de instalar PWS el servidor Web no funcionará.

    Precaución    Se producirán errores de instalación de ODBC durante el programa de instalación si hay cualquier otra aplicación o servicio del sistema en ejecución que utilice el software de ODBC existente en el sistema. Si ocurre algún error de ODBC, detenga todas las aplicaciones de escritorio y servicios del sistema, y ejecute de nuevo el programa de instalación.

  4. Para utilizar PWS en un equipo no conectado a una red, debe utilizar el siguiente procedimiento:
    1. Cambie el nombre de Hosts.sam, que se encuentra en el directorio \Windows a Hosts (sin extensión).
    2. Abra el archivo Hosts en un editor de textos. Agregue una entrada que asigne 127.0.0.1 al nombre del equipo y guarde los cambios.
    3. Haga clic en Inicio, seleccione Configuración y haga clic en Panel de control.
    4. Haga doble clic en el icono Red.
    5. En la hoja de propiedades Configuración, bajoEstán instalados los siguientes elementos de red seleccione Adaptador de acceso telefónico TCP/IP y haga clic en Propiedades.
    6. En la hoja de propiedades Direcciones IP, seleccione Especifique una dirección IP.
    7. En el cuadro Dirección IP, escriba 1.2.3.4.
    8. En el cuadro Máscara de subred, escriba 255.0.0.0.
    9. Haga clic en Aceptar.
    10. Es posible que necesite insertar el CD de Microsoft Windows 95 y, a continuación, reiniciar el equipo para que los cambios surtan efecto.

    Nota   Antes de llamar al proveedor de servicios Internet, debe restablecer la configuración en la hoja de propiedades Dirección IP a Obtener una dirección IP automáticamente.

Instalar.exe no tiene firma de código.
Al ejecutar la instalación desde el CD de PWS mediante la página Web de instalación, Internet Explorer mostrará un mensaje de error que indica que Instalar.exe no tiene firma de código. Instalar.exe es un archivo de 16 bits y los formatos de archivo de 16 bits no pueden tener firma de código.

Documentación del producto

Toda la documentación del producto está disponible en pantalla. Puede ver los temas de Ayuda si hace clic en Inicio y selecciona el grupo de programas donde está instalado el producto o mediante el menú Ayuda de la herramienta de administración. También hay disponible Ayuda contextual si presiona la tecla F1 mientras ve una hoja de propiedades o un cuadro de diálogo.

Utilizar la documentación

Tenga en cuenta las siguientes instrucciones al utilizar la documentación en pantalla:

  • Si hace clic en el icono Imprimir del marco Contenido de la documentación de Option Pack y selecciona la opción Imprimir todo lo que se encuentra bajo el título actual, es posible que se produzca una disminución momentánea del rendimiento del servidor si la sección seleccionada contiene un gran número de temas. (Por esta razón, la opción Imprimir todo lo que se encuentra en el contenido no está disponible en esta versión.)

    • Las páginas impresas mediante el icono de impresión de Contenido no incluirán gráficos ni documentos vinculados. Además, el icono de impresión de Contenido no funciona junto con las secuencias de comandos y los programas de ejemplo que se encuentran en la sección Ejemplos para el programador de la documentación de PWS. Si selecciona un tema de la sección Ejemplos para el programador y hace clic en el icono de impresión de Contenido, aparecerá un mensaje de error que indica que no se puede encontrar el archivo.

    Una forma alternativa de imprimir temas individuales es hacer clic con el botón secundario del mouse (ratón) en un tema y, a continuación, hacer clic en Imprimir.

  • Si Internet Explorer muestra un mensaje de error de HTTP 1.1/404 Object Not Found (Objeto no encontrado) cuando intenta ver la documentación, es posible que el archivo o los archivos que intenta ver no se hayan instalado. Para instalar el archivo o los archivos, vuelva a ejecutar el programa de instalación y elija la opción adecuada de documentación.

  • El servicio WWW debe estar instalado y se debe ejecutar el servidor para poder ver la documentación (con la excepción de las notas de la versión y los archivos de solución de problemas). Si el sitio Web está detenido, al hacer clic en Documentación del producto obtendrá el mensaje "No se ha podido establecer una conexión con el servidor". Si se produce este error, inicie Personal Web Manager y compruebe el estado del sitio Web; si está detenido, inicie el servicio.

  • La mayoría de la exploración por la documentación tiene lugar en el marco izquierdo. Para ver la tabla de contenido, haga clic en el botón Contenido. Haga clic en un encabezado para ver las páginas que contienen los resúmenes de la información incluida en esa sección del árbol del contenido. Expanda y contraiga las ramas del árbol de contenidos para ver u ocultar secciones de temas.

  • Para ver el índice, seleccione la hoja de propiedades en el marco izquierdo. Escriba una palabra para avanzar hasta la parte del índice más próxima a la palabra escrita. Haga doble clic en una palabra para ver el tema o la lista de temas que contienen la palabra clave.

    Nota   Según el explorador que utilice, los dispositivos de exploración del marco izquierdo son controles ActiveX™ (en Internet Explorer) o controles Java (en exploradores de Netscape); por tanto, si no ve la tabla de contenido en el marco izquierdo, asegúrese de que tiene la siguiente configuración de explorador. Para Internet Explorer, establezca la seguridad del explorador como Media y habilite los controles y complementos ActiveX. Para exploradores de Netscape, habilite Java y JavaScript.

  • Alguna funcionalidad disponible en el marco izquierdo de la documentación, como el botón de sincronización de la tabla de contenido (TOC), puede no funcionar correctamente si tiene instalada una versión anterior de Hhctrl.ocx. Como no se ha activado el control de versión para este archivo en la documentación, busque Hhctrl.ocx y elimine las copias del archivo del disco local. La próxima vez que tenga acceso a la documentación, se le pedirá que acepte el certificado que instalará el control TOC actualizado.

Otras limitaciones y problemas conocidos

La siguiente lista contiene problemas y limitaciones que se sabe que existen en esta versión:

  • Personal Web Server no es compatible con servicios FTP.
  • Si actualiza desde una versión anterior de PWS y tiene problemas para ver Default.asp, es probable que tenga una versión más antigua de este archivo. Para instalar la nueva versión PWS 4.0 de este archivo, debe quitar manualmente la versión antigua de Default.asp. La ubicación es C:\InetPub\Wwwroot.
  • En algunos casos, los archivos que quedan tras una instalación anterior de las Extensiones de servidor de FrontPage pueden evitar que las nuevas versiones de este componente se instalen correctamente. Si el programa de instalación de PWS informa de errores durante la instalación de las Extensiones de servidor de FrontPage o si PWS, Microsoft FrontPage o Microsoft Visual InterDev no pueden conectarse a un sitio Web de FrontPage, utilice el siguiente procedimiento para corregir el problema:
    1. Ejecute el programa de instalación de PWS, elija Agregar o quitar y desinstale las Extensiones de servidor de FrontPage.
    2. Elimine los siguientes archivos del directorio \Windows\System\: Fp30txt.dll, Fp30utl.dll y Fp30wel.dll.
    3. Elimine el directorio \Archivos de programa\Microsoft FrontPage\version3.0 incluidos todos los archivos y subdirectorios.
    4. Ejecute el programa de instalación de PWS y reinstale las Extensiones de servidor de FrontPage.
  • Los errores que ocurren durante la instalación desatendida se escribirán en el Registro de Windows 95. Vaya a la siguiente ruta de acceso del Registro para obtener información acerca del error:
    HKEY_LOCAL_MACHINE\
     Software

       \Windows

  • Ejecutar Personal Web Server 4.0 en un equipo con un procesador Intel 486 de velocidad inferior a 50 MHz y con 16 MB o menos de RAM aumenta la probabilidad de que los motores de secuencias de comandos se degraden a un modo de empaquetado de pilas de memoria, un proceso que evitará que el servidor acepte nuevas conexiones. El proceso de empaquetado de pilas puede tardar de 20 a 30 minutos en completarse (dependiendo de la velocidad del microprocesador).
  • Páginas Active Server
    • Si la única base de datos que utiliza con las páginas Active Server® (ASP) es Microsoft SQL Server&; es decir, si nunca utiliza archivos de Microsoft Access (.mdb) con ASP y crea instancias del componente Database Access que almacena en el objeto Session, puede mejorar el rendimiento si cambia en el Registro el modelo de subprocesamiento para las instancias del componente Database Access de Apartment (Controlado) a Both (Ambos). Para cambiar de Controlado a Ambos, haga doble clic en Makefre15.bat en la carpeta Archivos de programa\Archivos comunes\System\Ado.

      • Para invertir este proceso, (es decir, para cambiar el modelo de subprocesamiento de Ambos a Controlado), haga doble clic en Makeapt15.bat en la carpeta Archivos de programa\Archivos comunes\System\Ado.

      Nota   Recomendamos que no almacene instancias del componente Database Access en el objeto Session.

    • En la documentación de ASP se indica incorrectamente que el método MapPath no es compatible con la sintaxis de rutas de acceso relativas, como ./ruta de acceso o ../ruta de acceso. El método MapPath admite sintaxis de rutas de acceso relativas si la propiedad EnableParentPaths de la metabase está habilitada. Cuando la propiedad EnableParentPaths no está habilitada, utilizar el método MapPath con una ruta de acceso relativa produce un error.
    • El objeto Scripting.Dictionary está marcado erróneamente como procesamiento Ambos. Se debería marcar como subprocesamiento Controlado. Para cambiar esto, utilice el Editor del Registro para abrir la siguiente clave del Registro:

      • HKEY_CLASSES_ROOT\
        CLSID
         \{EE09B103-97E0-11CF-978F-00A02463E06F}
          \InprocServer32

      Cambie el valor con nombre para ThreadingModel a Apartment. Si utiliza el objeto Dictionary con alcance de aplicación sin hacer este cambio, pueden dañarse los datos.

    • Si cambia los enlaces de red en un sitio Web en ejecución con usuarios activos, estos quedarán desconectados.
    • Las propiedades establecidas para los objetos Request, Response y Server de ASP sólo son válidas durante el proceso de una petición. Por ejemplo, si establece una propiedad en el archivo Global.asa, como Response.Buffer = True, la propiedad sólo se aplicará a una única petición y no a todas las peticiones.
    • Actualmente hay problemas conocidos al utilizar PerlScript con ASP. Si ejecuta PerlScript de ActiveWare, póngase en contacto con ActiveWare para obtener un nuevo motor de PerlScript.
    • La documentación de Scripter's Reference enumera las propiedades del objeto MyInfo y afirma que Personal Web Server inicializa estas propiedades. Aunque esta información es válida para una instalación Macintosh de Personal Web Server en un equipo Macintosh, puede no ser válida para una instalación Windows 95 de Personal Web Server. No existe ninguna de las propiedades enumeradas en la documentación de Scripter's Reference cuando se crea por primera vez una instancia de MyInfo. En su lugar, estas propiedades se crean durante la ejecución mediante una instrucción de asignación. Esto significa que si una propiedad no existe, al consultar su valor se devolverá una cadena vacía y cualquier intento de utilizar la propiedad en una expresión devolverá un error.
    • Si crea o utiliza componentes desarrollados en Microsoft® Visual Basic5.0™ (Edición empresarial o Edición profesional) para utilizarlos con ASP, se recomienda encarecidamente que actualice a Visual Studio97 Service Pack 2, tanto para los equipos de programación como para los servidores.

      • Puede descargar Visual Studio 97 Service Pack 2 en http://www.microsoft.com/vstudio/sp/ (este sitio está en inglés).

  • Algunas características de acceso a redes de esta versión de Windows NT Internet Connection Services para el Servicio de acceso remoto de Microsoft se diseñaron para atender las necesidades específicas de los Operadores de redes públicas y de los Proveedores de servicios Internet (ISP). Estas características se lanzan en su totalidad para facilitar las pruebas de distribución e interoperabilidad, así como para garantizar la calidad de las implementaciones de los servicios al lanzarlos al mercado. Estas características centradas en ISP no estarán disponibles en la versión final de Windows NT Server. En su lugar, estas características estarán disponibles en una futura versión diseñada específicamente para Operadores de redes públicas e ISP. Para obtener más información acerca de estas nuevas tecnologías de VPN y para determinar que características específicas estarán disponibles en Windows NT Server, visite el sitio Web de Windows NT Communications que se encuentra en http://www.microsoft.com/communications/.

    • Por ejemplo, si publicdir es el nombre del recurso compartido en el equipo de origen al que puede tener acceso el equipo de destino, escribiría publicdir para el recurso compartido Duplicación.

  • Al ejecutar componentes de Java en un servidor ocupado, es posible que la máquina virtual (VM) de Java quede bloqueada durante la inicialización, como resultado de ser inicializada desde 40 o más subprocesos. Esto sólo ocurre la primera vez que se inicializa Java VM por proceso. Como solución, agregue el siguiente código al archivo Global.asa para la aplicación ("ObjetoJava" puede ser un componente ficticio que no haga nada).
    • 
<SCRIPT LANGUAGE=vbscript  RUNAT=SERVER>

Sub Application_OnStart
	set initobj = Server.CreateObject("ObjetoJava")

End Sub

</SCRIPT>
  • Sólo puede publicar archivos del disco duro local. Por razones de seguridad, el asistente para publicar de Personal Web Manager no publicará un archivo que resida en una unidad de red.
  • En el Asistente para página principal de Personal Web Manager, los gráficos de la hoja de estilos asociada a las plantillas de página principal se ven mejor con Internet Explorer 4.01. Las mínimas diferencias entre la forma en que aparece la plantilla de página principal en IE 3.0 y en Netscape son normales.
  • Internet Explorer versiones 3.0 y 4.0 pasa por alto los caracteres de salto de línea en campos de texto. Para agregar saltos de línea a una entrada de un campo de texto, agregue la etiqueta HTML <BR> al final de cada línea.
  • ActiveX Data Objects (ADO) utilizan el método estándar de COM para pasar información de errores pero ADO no comprueba si se ha colocado cualquier otra información en el objeto ErrorInfo y borra ese objeto cuando se destruye. Hay una solución mediante código a este problema en Microsoft® Visual Basic®, Visual C++® y Visual J++™ que debe implementar en el controlador de errores.

    • Para Visual Basic, el código debe decir:

    
ErrorHandler:
    ' limpieza
    If Not adoRS Is Nothing Then
	Set adoRS = Nothing
    End If
    If Not adoConn Is Nothing Then
	Set adoConn = Nothing
    End If
    
    Err.Raise Err.Number, "Bank.Accout.Post", Err.Description

Exit Function

    Para Visual C++, el código debe decir:

    
//
// ErrorInfo se guarda aquí porque el siguiente   
// código de limpieza de ADO puede borrarlo.
//
IErrorInfo * pErrorInfo = NULL;
GetErrorInfo(NULL, &pErrorInfo);

if (adoRsBalance) adoRsBalance->Release();
if (adoCoConnection) adoCoConnection->Release();

AtlReportError( CLSID_CAccount, pErrMsg, IID_IAccount, hr);

//
// volver a colocar el error en TLS
//
SetErrorInfo(NULL, pErrorInfo);

    Para Visual J++, el código debe decir:

    
if (adoRsBalance != null) {
	if (adoRsBalance.getState() == ObjectStateEnum.adStateOpen)
		adoRsBalance.Close();
	ComLib.release (adoRsBalance);
}

if (adoConn != null) {
	if (adoConn.getState() == ObjectStateEnum.adStateOpen)
		adoConn.Close();
	ComLib.release (adoConn);
}

    Nota   En Visual J++ debe cerrar explícitamente los objetos Recordset o las conexiones, así como liberar explícitamente los objetos de ADO.

  • Remote Data Service (RDS) 1.5 no funciona cuando está instalado como servidor en Windows 95 o Windows 98 Beta  que utiliza Personal Web Server. Esto se solucionará en una versión posterior del Remote Data Service. Actualmente, RDS se ejecuta correctamente desde Windows NT Server o Workstation, en plataformas x86 o Alpha.
  • Si escribe componentes de servidor en Java y desea emplear Microsoft®  Visual J++™ 1.1 para depurar los componentes, utilice el siguiente procedimiento:
    1. Genere archivos .class de depuración para las clases de Java que desea depurar.
    2. Asegúrese de que la aplicación de PWS que llama al componente Java está configurada para ejecutarse en proceso (la configuración predeterminada). Este mecanismo de depuración no funciona con aplicaciones fuera de proceso.
    3. En Visual J++, abra el proyecto que contiene las clases Java, haga clic en el menú Proyecto, haga clic en Configuración y, a continuación, haga clic en la hoja de propiedades Depuración.
    4. En la lista Categoría, seleccione General.
    5. En el cuadro Clase para depurar o ejecutar , escriba el nombre completo (incluido el nombre del paquete) de la clase que desea depurar. No incluya una extensión de archivo. Por ejemplo, puede escribir "IISSample.HolaATodos" como nombre de clase; Visual J ++ distingue mayúsculas de minúsculas.
    6. Bajo Depurar o ejecutar proyecto bajo, seleccione Intérprete independiente (sólo aplicaciones).
    7. En la lista Categoría, seleccione Intérprete independiente . Bajo Intérprete independiente, escriba la ruta de acceso completa al archivo Inetinfo.exe en el sistema. Por ejemplo, si es aplicable puede escribir C:\Winnt\System32\Inetsrv\Inetinfo.exe, que es la ubicación predeterminada para Inetinfo.exe.
    8. Bajo Argumentos del intérprete independiente escriba -e w3svc.

      9. Nota   Deje en blanco el cuadro Argumentos de programa ; de lo contrario, el procedimiento no funcionará.

    9. Haga clic en Aceptar.
    10. En Visual J++, establezca un punto de interrupción en la clase Java.
    11. En el menú Generar, elija Comenzar a depurar y haga clic en Ejecutar.
      12. Este paso iniciará Inetinfo.exe como un proceso.

      Nota   Al seleccionar Ejecutar <nombre_clase> no se iniciará la depuración.

    12. Después de que se inicie el depurador de Visual J++, invoque a la clase Java; para ello, abra el archivo .asp que invoca a la clase. Tiene aproximadamente 30 segundos para invocar el código antes de que el depurador de Visual J++ alcance el límite de tiempo de espera. Cuando el depurador llegue al punto de interrupción, Visual J++ ya no exigirá el límite de tiempo de espera, así que puede continuar depurando el componente.
    13. La documentación en línea del producto contiene guías de referencia para programadores y componentes SDK, ambos en inglés.
    14. Si tiene problemas al imprimir la documentación en línea puede ser debido a la versión actual de un control ActiveX. Actualice su explorador a Internet Explorer 4 Service Pack 1.
